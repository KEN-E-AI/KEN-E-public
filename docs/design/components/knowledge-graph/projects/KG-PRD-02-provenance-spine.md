# KG-PRD-02 — Provenance Spine: `Session`, `Observation`, `ResearchRun` Nodes + GraphSyncService Methods

**Status:** Blocked — resumes once KG-PRD-01, DM-PRD-00, and DM-PRD-05 ship
**Owner team:** Backend (foundation)
**Blocked by:** KG-PRD-01, DM-PRD-00 (Shape B convention for the three new subcollections), DM-PRD-05 (recursive-delete coverage)
**Blocks:** KG-PRDs 3, 4, 5
**Estimated effort:** 3–4 days

---

## 1. Context

Today the Neo4j KB has no concept of *who added a fact*, *when it was observed*, *whether it is still true*, or *which session surfaced it*. The 28 strategy node types track `created_time` and `created_by` (a user_id), but that's it. There is no way to ask "what was learned in session X?" or "which facts in this KB were auto-generated by research vs. surfaced in conversation?"

This PRD adds the provenance spine that every downstream KG feature relies on:

- Three new node types (`Session`, `Observation`, `ResearchRun`) that describe *episodes* of knowledge acquisition.
- Six new relationship types that link facts back to those episodes.
- Bi-temporal properties (`valid_from`, `valid_to`, `superseded_by`) so a fact can be retired without being deleted.
- Extensions to the existing `GraphSyncService` so every write stamps provenance automatically.

The per-turn `touch_session` hook that KG-PRD-04 relies on is also delivered here so `Session` nodes come into existence lazily the moment a conversation starts writing.

## 2. Scope

### In scope
- New node types: `Session`, `Observation`, `ResearchRun`. Registered in `NODE_TYPE_REGISTRY` with prefixes.
- New relationship types (written by this PRD's service methods): `OBSERVED_IN`, `ESTABLISHED_BY`, `UPDATED_BY`, `ABOUT`, `SUPERSEDES`, `FOR_ACCOUNT`.
- New bi-temporal / provenance properties on every `:KGNode`: `valid_from`, `valid_to`, `source_session_id`, `source_research_run_id`, `last_updated_by_agent`.
- `GraphSyncService` methods: session lifecycle (`create_session` / `touch_session` / `close_session`), observation lifecycle (`create_observation` / `supersede_observation`), research-run lifecycle (`create_research_run` / `close_research_run`).
- Automatic provenance stamping inside `create_node` / `update_node`: when a caller passes `session_id` or `research_run_id`, the service attaches `:OBSERVED_IN` / `:ESTABLISHED_BY` / `:UPDATED_BY` edges and writes `source_session_id` / `source_research_run_id` properties.
- Per-turn `touch_session` hook in the chat router so `Session` nodes are created lazily.
- REST endpoints for `Session` and `Observation` (CRUD via the existing `crud_factory` pattern).
- Unit + integration tests.

### Out of scope
- Any `Proposal` node (proposals live in the Automations platform — see KG-PRD-04).
- Migration of existing data to backfill `source_session_id` / `source_research_run_id` on pre-existing nodes (they simply retain `null`).
- Vector embeddings for `Observation` nodes — KG-PRD-03 owns embedding generation for read-tool use, and KG-PRD-04 owns the writer that generates embeddings for new observations.
- Orchestrator-side read tools — KG-PRD-03.

## 4. Data contract

### `Session` — one per ADK chat session

```python
class Session(KGNodeBase):
    session_id: str                          # matches ADK's session id
    node_id: str                             # == session_id for simplicity
    account_id: str
    user_id: str
    started_at: datetime                     # first turn
    last_message_at: datetime                # updated every turn
    ended_at: datetime | None = None         # set by close_session
    message_count: int = 0
    summary: str | None = None               # written by KG-PRD-04's reviewer agent
    status: Literal["active", "processing", "processed", "failed"] = "active"
```

Node label: `:Session:KGNode`. Constraint: `session_id_unique` (migration 001 already creates it).

### `Observation` — free-form conversational fact

```python
class Observation(KGNodeBase):
    node_id: str                             # "obs_<account-id>_<8-hex>"
    account_id: str
    subject: str                             # short noun phrase, indexed, embedded along with statement
    statement: str                           # the fact itself — full sentence
    confidence: Literal["high", "medium", "low"]
    embedding: list[float] | None = None     # 768-dim; populated by KG-PRD-04 writer
    valid_from: datetime                     # defaults to now()
    valid_to: datetime | None = None         # null = current; set when superseded
    superseded_by: str | None = None         # node_id of the newer Observation
    # audit (inherited from KGNodeBase)
    created_time: datetime
    last_modified: datetime
    created_by: str                          # user_id or agent id
    last_modified_by: str
    # provenance (inherited)
    source_session_id: str | None            # always set — observations are session-born
    source_research_run_id: str | None = None
    last_updated_by_agent: Literal["researcher", "session_end_agent", "user"]
```

Node label: `:Observation:KGNode`. Prefix: `obs`. No `max_per_account` limit.

### `ResearchRun` — one per initial account research pass

```python
class ResearchRun(KGNodeBase):
    run_id: str                              # UUID
    node_id: str                             # == run_id
    account_id: str
    started_at: datetime
    ended_at: datetime | None = None
    status: Literal["running", "complete", "failed"]
    agents: list[str]                        # ["business", "competitive", "marketing", "brand"]
```

Node label: `:ResearchRun:KGNode`. Prefix: `research`.

### Relationships (written by this PRD's service methods)

| Pattern | Written when | Written by |
|---|---|---|
| `(:KGNode)-[:OBSERVED_IN]->(:Session)` | Any write where `session_id` is supplied | `create_node`, `update_node`, `create_observation` |
| `(:KGNode)-[:ESTABLISHED_BY]->(:ResearchRun)` | Any write where `research_run_id` is supplied | `create_node` |
| `(:KGNode)-[:UPDATED_BY]->(:Session)` | `update_node` call where `session_id` is supplied | `update_node` |
| `(:Observation)-[:ABOUT]->(:KGNode)` | `create_observation` with `about_node_id` | `create_observation` |
| `(:Observation)-[:SUPERSEDES]->(:Observation)` | `supersede_observation` | `supersede_observation` |
| `(:Session)-[:FOR_ACCOUNT]->(:Account)` | `create_session` | `create_session` |

### Bi-temporal / provenance properties on every `:KGNode`

These exist on every node type (the 28 existing strategy types + the new three). Migration 002 (KG-PRD-01) already backfilled `valid_from`. KG-PRD-2 writes the rest on new writes and future updates.

```
valid_from: datetime                        # set by create_node
valid_to: datetime | None                   # set only by soft-delete / supersede
source_session_id: str | None               # set when session_id supplied on write
source_research_run_id: str | None          # set when research_run_id supplied on write
last_updated_by_agent: Literal[             # set on every write
  "researcher", "session_end_agent", "user"
]
```

The `last_updated_by_agent` value is derived from caller context (the router / service invoking `GraphSyncService`):
- `"researcher"` → any write originating from strategy-agent builders (KG-PRD-05).
- `"session_end_agent"` → writes from KG-PRD-04's applier.
- `"user"` → every other write (direct CRUD endpoint from the frontend).

### Service method shapes

```python
# api/src/kene_api/services/graph_sync_service.py

async def create_session(
    self,
    session_id: str,
    account_id: str,
    user_id: str,
    now: datetime | None = None,
) -> Session:
    """Idempotent — MERGE on session_id. Creates Session + :FOR_ACCOUNT edge."""

async def touch_session(
    self,
    session_id: str,
    account_id: str,
    user_id: str,
    now: datetime | None = None,
) -> None:
    """Create (lazy) + bump last_message_at + increment message_count by 1.
    Single MERGE transaction; must be safe to call on every chat turn."""

async def close_session(
    self,
    session_id: str,
    status: Literal["processed", "failed"],
    summary: str | None = None,
    ended_at: datetime | None = None,
) -> None:
    """Sets ended_at + status + summary. Called by KG-PRD-04's applier on success/failure."""

async def create_observation(
    self,
    account_id: str,
    subject: str,
    statement: str,
    confidence: Literal["high", "medium", "low"],
    session_id: str,                         # always required — observations are session-born
    about_node_id: str | None = None,
    embedding: list[float] | None = None,
    created_by: str = "session_end_agent",
) -> Observation:
    """Creates Observation + :OBSERVED_IN edge; :ABOUT edge if about_node_id provided.
    Stamps source_session_id, last_updated_by_agent, valid_from."""

async def supersede_observation(
    self,
    old_node_id: str,
    new_statement: str,
    session_id: str,
    account_id: str,
    confidence: Literal["high", "medium", "low"] | None = None,
    created_by: str = "session_end_agent",
) -> Observation:
    """Creates a new Observation with the same subject/about/confidence as the old one
    (confidence overrideable), then in the same transaction sets old.valid_to = now()
    and writes (:new)-[:SUPERSEDES]->(:old) plus old.superseded_by = new.node_id."""

async def create_research_run(
    self,
    account_id: str,
    agents: list[str],
    now: datetime | None = None,
) -> ResearchRun:
    """Creates ResearchRun node. Returns the model with run_id populated."""

async def close_research_run(
    self,
    run_id: str,
    status: Literal["complete", "failed"],
) -> None:
    """Sets ended_at + status."""
```

### Provenance stamping in `create_node` / `update_node`

Both methods gain an optional `session_id: str | None = None` and `research_run_id: str | None = None` parameter. When present:

- `create_node`:
  - Write `source_session_id` / `source_research_run_id` as node properties.
  - After the node is created (and *after* the Firestore sync), execute a small follow-up query:
    ```cypher
    MATCH (n {node_id: $node_id, account_id: $account_id})
    OPTIONAL MATCH (s:Session {session_id: $session_id})
    OPTIONAL MATCH (r:ResearchRun {run_id: $run_id})
    FOREACH (_ IN CASE WHEN s IS NOT NULL THEN [1] ELSE [] END |
      MERGE (n)-[:OBSERVED_IN]->(s)
    )
    FOREACH (_ IN CASE WHEN r IS NOT NULL THEN [1] ELSE [] END |
      MERGE (n)-[:ESTABLISHED_BY]->(r)
    )
    ```
  - Also stamp `last_updated_by_agent` from an explicit caller parameter (defaulting to `"user"` when absent).

- `update_node`: same pattern, but `:UPDATED_BY` instead of `:OBSERVED_IN`, and only if `session_id` is supplied.

- Rollback consideration: the existing `create_node` rolls back the Neo4j create if the Firestore sync fails. The provenance edge is written *after* Firestore sync succeeds, so there's no rollback path needed for the edges — if Neo4j+Firestore succeed but edge-write fails, the node is still consistent and a subsequent retry can re-write the edges (they use `MERGE`). Log a warning but don't 500 the request.

### Per-turn `touch_session` hook

In `api/src/kene_api/routers/chat.py`, inside the chat message handler that processes a single turn:

```python
# After validating auth + extracting session_id / account_id / user_id:
await graph_sync_service.touch_session(
    session_id=session_id,
    account_id=account_id,
    user_id=user_id,
)
```

Behavior:
- Fire-and-forget? **No** — we want the write to complete before the agent runs so that downstream tools (KG-PRD-03 / KG-PRD-04) can assume the Session node exists. The write is a single Cypher `MERGE` — sub-20ms typical — so adding it synchronously is acceptable.
- Failure handling: if `touch_session` raises, log + continue. A missed touch doesn't block the chat turn; KG-PRD-04's sweeper will still process sessions that exist but missed touches (it queries on `last_message_at`, which will be slightly stale but still finite).

### Validation service updates

- Whitelist new node types in `VALID_NODE_TYPES`.
- Per-type validators:
  - `Session`: `session_id` matches the account's accessible sessions for the acting user. (When invoked from the per-turn hook, this is trivially true.)
  - `Observation`: `subject` non-empty and <= 200 chars; `statement` non-empty and <= 2000 chars; if `about_node_id` provided, that node must exist and belong to the same account.
  - `ResearchRun`: `agents` non-empty; each entry from the known set.

## 7. Acceptance criteria

1. Registering the three new node types in `NODE_TYPE_REGISTRY` does not break the existing `crud_factory` — all 28 legacy routers still resolve and their tests pass unchanged.
2. Creating a Session via `touch_session` twice in a row produces one Session node with `message_count=2`.
3. `touch_session` is called from the per-turn chat handler; a new chat conversation produces a Session node with `status="active"` and a `:FOR_ACCOUNT` edge to the expected Account.
4. Creating a strategy node (e.g. `Competitor`) through the existing CRUD endpoint with a `session_id` query param results in: the node carrying `source_session_id`, `last_updated_by_agent="user"`, and an `:OBSERVED_IN` edge to that Session.
5. Updating the same node with a different `session_id` adds an `:UPDATED_BY` edge (without removing the original `:OBSERVED_IN`).
6. Creating an Observation with `about_node_id` pointing at an existing Competitor adds an `:ABOUT` edge; omitting `about_node_id` leaves only the `:OBSERVED_IN` edge.
7. Superseding an Observation: `supersede_observation` creates the new node, sets `old.valid_to` and `old.superseded_by`, and writes `(:new)-[:SUPERSEDES]->(:old)` — all within one transaction (verified by simulating a mid-transaction failure and observing that neither side is written).
8. `close_session` with `status="processed"` and a summary string persists both; re-calling it is idempotent (same-or-later `ended_at`).
9. `create_research_run` returns a `run_id`; closing it sets the fields and leaves every node it produced linked via `:ESTABLISHED_BY` (spot-check via a fixture).
10. Cross-account reads on any endpoint return `403`. Cross-account `about_node_id` on observation create returns `422` via the validation service.
11. All tests pass; `make lint` clean.

## 9. Risks & open questions

| Risk / question | Mitigation |
|---|---|
| Per-turn `touch_session` adds latency to chat | Target < 20ms. If it regresses, gate behind a feature flag and measure. Failing loudly is preferred over silent drift; any measurable regression should be visible in Weave traces. |
| Provenance edge writes after Firestore sync — a three-legged transaction | Accepted: edges use `MERGE`, retry-safe. A failure leaves properties-only provenance (`source_session_id`) without the edge. KG-PRD-04's tools MUST query by property (`WHERE n.source_session_id = $id`), not by edge, as the canonical source of truth. Document. |
| `last_updated_by_agent` value passed from the router → service — easy to forget | Default parameter is `"user"`. Agent-originated calls (KG-PRD-04, KG-PRD-05) must pass their own value explicitly. Make the test fixtures use a non-default to catch drift. |
| Observation writes without `session_id` | Reject at the service level — `session_id` is required (a conversation fact has no other home). If a future PRD needs schema-less observations, widen then. |
| Session node bloat — one per chat turn worth of metadata | `message_count` increments but payload stays tiny. No expected issue. |
| `ResearchRun` agents list drift | Keep the known set in `constants.py`; validator rejects unknown entries. |
| Legacy strategy nodes created before this PRD have no `source_session_id` or `last_updated_by_agent` | Fine — they keep `null`. KG-PRD-04's routing logic tolerates null (treats it as "not user-written, safe to auto-apply"). |
